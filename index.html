业务状态流转

TASK-013 — 月台作业完整状态机（含 PLC 安全联锁节点）

① 等待识别 HMI 显示等待人脸刷卡界面；人脸 ISUP 独立进程持续监听，事件经 HTTP → 后端 → WebSocket 推送到对应月台 HMI。
② 身份确认 后端校验人脸事件与当前任务操作工绑定关系；操作工权限验证通过，页面切换到作业信息页。
③ 任务接收 后端向 AGV 调度系统请求任务；调度响应后写入 PostgreSQL，推送任务信息到 HMI；操作工确认接单。
④ 装车控制 HMI 进入设备控制页；操作工触发装车指令，后端通过 PLC 排队锁写入 SLMP 指令；PLC 物理联锁检查到位信号后开始放料。
⑤ 暂停 / 换人 主管可触发暂停，当前任务挂起，PLC 联锁保持；换人需重新人脸验证后方可恢复；状态持久化到 PostgreSQL 防止断线丢失。
⑥ 结束装车 操作工确认结束；后端向调度回报完成，写入完工记录；PLC 联锁解除，月台状态归零；HMI 返回等待识别页。
⑦ 异常处理 任一环节异常（PLC 超时、调度失联、WebSocket 断线）→ 后端记录报警 → HMI 报警页显示 → 主管介入处理 → 恢复或终止任务。
A / B 机型差异：A 类（1#、12# 月台）为端点机位，具备额外的 LED 大屏控制权限；B 类（2#–11#）为标准机位。角色分层：操作工极简界面 / 主管工业界面 / 管理员调试界面，三层权限由后端 token 验证控制。

版本发布与回滚

TASK-013 — .wrelease 全网协同部署流程

① 构建 .wrelease CI 流程打包前端静态文件为 .wrelease 格式包，包含版本号、构建时间戳和完整性哈希。
② 包完整性校验 部署控制器导入 .wrelease，校验哈希与包结构；任一校验失败，拒绝继续，不下发任何指令。
③ 业务就绪检查 控制器调用 /api/system/deployment-readiness，逐一检查 01–12 号月台：当前任务为空、状态为 idle、无阻断报警。任一不满足则拒绝发布。
④ 全节点准备下发 控制器向 12 台 HMI Agent 下发"准备"指令；每台 Agent 拉取包、解压、校验、预加载到待激活目录；完成后回报 ready。
⑤ 统一激活（60s 时限） 全部 12 台回报 ready 后，控制器下发统一激活指令；每台 Agent 执行 Nginx 配置切换 + 重启 Kiosk；60 秒内需全部完成健康确认。
⑥ 健康检查 控制器轮询各节点 /health 接口，确认新版本正在服务；全部通过则部署完成，新版本写入 current-version.json。
⑦ 失败全网回滚 任一节点在 60s 内未回报成功 → 控制器立即下发全网回滚指令 → 所有 Agent 切回前一版本目录（物理保留，永久禁止删除）→ 重启 Nginx → 状态上报。最坏情况中断时间 < 90 秒。
HMI Bootstrap 兜底：以 SYSTEM 身份随系统启动，每 30 秒检查 Agent 文件、配置、进程和计划任务；Agent 损坏或缺失时从 C:\WeiliHmi\bootstrap\payload 本地可信副本自动恢复，不依赖网络或控制器。

异常自愈链路

TASK-013 — 两级守护 + 三类隔离机制

第一级：NSSM 服务守护
  • AppExit Default Restart：进程退出即重启
  • AppRestartDelay=5000：崩溃后 5 秒拉起
  • AppThrottle=1500：秒崩检测，触发节流惩罚
  • 三项服务：业务后端 / 人脸服务 / 部署控制器
第二级：控制器主动守护
  • 每 10–30s 执行 sc.exe query 查询服务状态
  • 非发版窗口内检测到 STOPPED 立即 sc.exe start
  • 弥补 NSSM Throttle 失效导致的守护断链
  • 形成双保险：NSSM 失效 → 控制器兜底
服务崩溃 Node.js / Python 进程异常退出 → NSSM 检测到退出 → 5 秒后重启 → 若秒崩则 Throttle 倍增延迟 → 控制器守护定时器在 10–30s 内主动拉起。
HMI 掉线 工业现场电磁干扰导致网络瞬时抖动 → HMI WebSocket 断线 → 指数退避重连（避免 12 台同时重连雪崩）→ 重连后强制拉取 PostgreSQL 最新状态快照。
WebSocket 断开 后端 WebSocket 连接丢失 → 前端触发指数退避重连机制 → 重连成功后后端推送全量状态恢复包 → 前端状态完全恢复，不依赖本地缓存。
发布包异常 .wrelease 哈希校验失败 / 节点健康检查失败 → 全网回滚到前一版本 → 前一版本物理目录永久保留禁止删除 → 90 秒内恢复稳定状态。
配置缺失 PLC 点位配置 / 人脸服务配置缺失 → 后端启动阶段校验失败 → 拒绝启动并输出明确错误日志 → 问题在上线前被拦截，不进入运行状态。
SDK 进程隔离：海康人脸 ISUP 5.0（C++ 动态库）以独立进程运行，通过 IPC/HTTP 与主进程通信。SDK 段错误只影响人脸进程，不株连 Node.js 主进程，其余 12 台月台的 PLC 轮询和 HMI 连接不受影响。

Engineering Q&A

关键问题与解决方案

这些是现场联调和系统设计中真实踩过的坑，不是包装出来的。

Q1：HMI 多终端状态不一致怎么处理？

问题 工业现场电磁干扰频繁，12 台 HMI 可能在不同时间点断线重连，断线期间 PLC 状态发生变更，重连后前端可能回滚到历史错误状态，误导操作工。 处理 后端以 PostgreSQL 持久化所有月台状态快照（DockStore 状态总线）；HMI 重连时强制拉取最新全量状态快照覆盖本地状态；WebSocket 采用指数退避重连（避免 12 台同时重连冲垮服务）。 结果 重连后界面立即恢复最新状态，不依赖客户端本地缓存，消除状态漂移。

Q2：人脸事件来了但前端没反应怎么定位？

问题 海康 ISUP 5.0 SDK 为 C++ 动态库，若直接加载到 Node.js 主进程，SDK 段错误会导致整个后端崩溃，所有月台同时中断。 处理 人脸服务以独立进程（C#/Python）运行，通过 IPC/HTTP 与 Node.js 主进程通信；SDK 崩溃只影响人脸进程，主进程继续运行；在调试控制中心加入人脸链路连通性测试页签，可逐段排查"SDK → 独立进程 → HTTP → 后端 → WebSocket → 前端"各节点状态。 结果 SDK 崩溃不株连主进程；链路排查时间从数小时缩短到分钟级定位。

Q3：PLC 配置缺失时为什么要拒绝启动？

问题 三菱 FX5U PLC 内部以太网控制芯片并发连接数极其有限；若配置缺失后端仍启动并轮询，会产生无效并发 TCP 连接，导致 PLC 通讯端口过载，进而造成写指令丢失，引发安全事故。 处理 后端启动阶段对 PLC 点位映射表（260512_PLC点位映射表.csv）进行完整性校验；配置缺失或格式错误时拒绝启动并输出明确错误日志，不进入监听状态；同时设计 PLC 独占排队锁，保证读写串行，防止并发冲突。 结果 配置错误在上线前即被拦截；生产环境中 PLC 通讯冲突次数降为零。

Q4：HMI 版本升级失败如何回滚？

问题 12 台 HMI 统一升级时，若部分节点激活失败，系统将处于新旧版本混跑状态，可能导致前后端接口不兼容，引发全线异常。 处理 .wrelease 包导入后先进行完整性校验；控制器向所有节点下发"准备"指令，全部 12 台回报就绪后才执行统一激活；60s 内任一节点健康检查失败，控制器立即触发全网回滚指令，各 HMI Agent 切回前一版本目录（物理保留，永久禁止删除）并重启 Nginx。 结果 回滚完成后全网一致恢复上一稳定版本，最坏情况下中断时间不超过 90 秒。

Q5：生产环境为什么要禁用 Mock？

问题 Mock 适配器在开发阶段模拟 PLC/调度/人脸响应，但若生产环境误用 Mock，工人操作的是虚假状态，PLC 实际动作与界面不符，存在严重安全风险。 处理 引入 apiMode / mockUiEnabled 双开关解耦；生产配置文件（deployment_control/native/config）强制锁定 apiMode=real，缺失 real adapter 配置时后端拒绝启动；Mock UI 仅允许在调试控制中心页签内启用，不影响主业务链路。 结果 生产链路与调试链路完全隔离，现场联调时可保留 Mock UI 调试界面的同时接入真实后端。
